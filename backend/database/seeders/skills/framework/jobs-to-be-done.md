# Jobs To Be Done

Uncover user jobs using Clayton Christensen's Jobs-to-be-Done framework. Transform product or feature context into actionable job statements.

**Input:** Product/feature context, user research, or problem statement
**Output:** JTBD analysis with job statements across all three dimensions

---

## The Methodology

**Source:** Clayton Christensen & Bob Moesta, _Competing Against Luck_ (2016)

Core principle: People don't buy products—they hire them to get a job done. Understanding the job reveals what truly drives decisions.

### Job Types

| Type           | Question                                 | Example                      |
| -------------- | ---------------------------------------- | ---------------------------- |
| **Functional** | What task are they trying to accomplish? | "Get from A to B quickly"    |
| **Emotional**  | How do they want to feel?                | "Feel in control of my time" |
| **Social**     | How do they want to be perceived?        | "Be seen as responsible"     |

### Job Statement Format

```
When [situation], I want to [motivation], so I can [outcome].
```

The situation provides context. The motivation is the job. The outcome reveals the deeper goal.

---

## Process

### 1. Establish Context

Identify the triggering situation:

- What circumstance prompts the need?
- What's the user's current state?
- What constraints exist?

### 2. Uncover Functional Jobs

Ask: "What is the user literally trying to accomplish?"

Probe:

- What task triggers this need?
- What steps are they taking today?
- What would "done" look like?

### 3. Uncover Emotional Jobs

Ask: "How does the user want to feel during and after?"

Probe:

- What frustrations do current solutions cause?
- What would relief or success feel like?
- What anxieties exist around this task?

### 4. Uncover Social Jobs

Ask: "How does the user want to be perceived by others?"

Probe:

- Who sees them doing this?
- What image do they want to project?
- What social consequences exist for failure?

### 5. Write Job Statements

For each job type, write statements using the format:

**Functional:**

> When [situation], I want to [accomplish X], so I can [functional outcome].

**Emotional:**

> When [situation], I want to [feel Y], so I can [emotional outcome].

**Social:**

> When [situation], I want to [appear Z], so I can [social outcome].

### 6. Identify Gaps

Compare jobs to current solutions:

- What jobs are underserved?
- What jobs are over-served?
- Where do current solutions create friction?

## Example Job Statements

**Context:** Professional scheduling a meeting with external stakeholders

**Functional:**

> When I need to coordinate a meeting with multiple external parties, I want to find a time that works for everyone, so I can move the project forward.

**Emotional:**

> When I'm going back and forth on scheduling, I want to feel like I'm not wasting anyone's time, so I can stay confident about the interaction.

**Social:**

> When I send calendar invites to executives, I want to appear organized and respectful of their time, so I can maintain credibility.

**Gap identified:** Current solution (email chains) serves functional job but creates emotional friction—anxiety about appearing disorganized.

## Common Mistakes

| Mistake                 | Problem                        | Fix                                     |
| ----------------------- | ------------------------------ | --------------------------------------- |
| Jobs that are solutions | "I want to use a calendar app" | Ask "why?" until you reach the real job |
| Too broad               | "I want to be productive"      | Add situational specificity             |
| Missing job types       | Only functional jobs listed    | Explicitly probe emotional and social   |
| Company perspective     | "Users want our product to..." | Rewrite from user's point of view       |

---

## Quality Checks

- [ ] Jobs describe outcomes, not solutions
- [ ] All three job types (functional, emotional, social) are addressed
- [ ] Situation context is specific, not generic
- [ ] Jobs are written from user perspective, not company perspective
- [ ] Job statements follow the "When... I want to... so I can..." format
- [ ] At least one gap between current solutions and jobs is identified
